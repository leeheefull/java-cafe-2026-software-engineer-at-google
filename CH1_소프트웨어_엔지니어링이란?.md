# 1주차: 소프트웨어 엔지니어링이란?

## 핵심 정의

> **소프트웨어 엔지니어링 = 프로그래밍 + 시간 + 규모 + 트레이드오프**

구글 내부 정의: "흐르는 시간 위에서 순간순간의 프로그래밍을 모두 합산한 것"

---

## 1.1 시간과 변경

- 코드의 **기대 수명**이 길수록 변경 대응 능력이 중요해짐
- 수명이 짧으면 "그냥 프로그래밍", 길면 "소프트웨어 엔지니어링"
- **지속 가능성**: 소프트웨어 생애 동안 필요한 모든 변경에 대응할 수 있는 역량
- **하이럼의 법칙**: API 사용자가 충분히 많으면 명세에 없는 동작까지 누군가 의존하게 됨 → 모든 변경은 누군가의 워크플로와 충돌
- "변하지 않기"를 목표로 하면 안 됨 — 기반 기술(보안, 하드웨어, 효율)은 계속 바뀌므로

## 1.2 규모 확장과 효율성

- 조직이 커지면 **비용이 선형보다 완만하게** 증가하는 정책이 필요 (확장 가능한 정책)
- **확장 어려운 정책 예시**: 폐기 시 모든 사용자에게 마이그레이션 떠넘기기, 개발 브랜치 난립
- **확장 가능한 정책 예시**:
  - **갈아타기 규칙(Churn Rule)**: 인프라팀이 직접 마이그레이션 책임
  - **비욘세 규칙**: "좋아했다면 CI 테스트를 준비해뒀어야지" — CI에 없는 테스트는 인프라팀 책임 아님
- **원점 회귀(Shift Left)**: 문제를 개발 프로세스 초기에 발견할수록 수정 비용이 급격히 줄어듦

## 1.3 트레이드오프와 비용

- 비용 = 금융 + 리소스(CPU) + 인적(엔지니어링 노력) + 거래 + 기회 + 사회적 비용
- 좋은 의사결정 = 근거 자료에 적절한 가중치를 부여한 판단
- **"내가 시켰으니까"는 안 됨** — 합의 도출이 목표
- 의사결정 요인: (1) 반드시 해야 하는 일(법적/고객 요구) (2) 근거 기반 최선의 선택

---

**한 줄 요약**: 프로그래밍이 "코드를 만드는 것"이라면, 소프트웨어 엔지니어링은 "그 코드가 시간과 규모 속에서 살아남게 하는 것"이다.

## "프로그래밍" vs "소프트웨어 엔지니어링"

| | 프로그래밍 | 소프트웨어 엔지니어링 |
|---|---|---|
| **시간** | 지금 동작하면 끝 | 수년간 변경 가능해야 함 |
| **규모** | 개인 | 팀/조직 |
| **핵심** | "어떻게 만들지?" | "어떻게 유지·진화시킬지?" |

→ 차이를 만드는 건 코딩 능력이 아니라 **변경 비용을 관리하는 판단력**

---

## AI가 프로그래밍을 대체하면, 엔지니어링은?

AI가 코드 생성을 가져갈수록 CH1의 세 축이 오히려 더 중요해진다.

- **시간**: AI는 "지금 동작하는 코드"를 만든다. 하이럼의 법칙은 AI 코드에도 적용됨 → 무엇을 바꾸고 지킬지 판단은 인간의 몫
- **규모**: AI가 코드를 쏟아낼수록 CI/CD, 폐기 전략, 정책 설계 등 규모 통제 구조의 필요성 증가
- **트레이드오프**: 비용의 가중치를 정하는 맥락(법적 요구, 조직 문화, 기회비용)은 사업 이해에서 나옴

> AI가 "프로그래밍"을 가져가면, 남는 것은 **"시간 + 규모 + 트레이드오프"** — 즉 엔지니어링 그 자체다.
